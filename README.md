# Patterns
What is a pattern?
- Pattern is an answer for a problem. A blueprint to solve a common problem.

Learning the patterns is a good source for understanding common problems, 
algorithms and developing routine.

they teach you to recognition common problems and apply already known solutions.

But how can you learn pattern but never been introduced to a problem it solves? 
I think theres no need learning and memorizing patterns.
Its better to see and even more, face those problems yourself.
It should teach you that the same problem can have many forms as well as 
solution to a problem can be expressed in any paradigms at any level of abstraction 
in any language.

Some patterns rise from the language unable to do something straightforward and a pattern
is just work around for programmer, i learned it when was reading a lot of patterns and ideas born
in Java.

Some patterns solve code complexity, some code duplication, some help you to reuse stuff and build
your understanding how to properly structure your code.

Pattern can have specific directions, for example distributed patterns help to manage 
distributed systems, and concurrent patterns help to handle concurrent computations.
There are many such categories like async, distributed, concurrent, domain specific,
and even more abstract like structured, behavioral, or creational patterns as GoF teaches.

Patterns usually relate to each other and can form groups.
Some of them usually work together, you will frequently see that one pattern uses another, or
they cooperate.
There are standalone patterns as well.

But as good to learn patterns, there is an absolute need to learn anti-patterns as well,
they teach you common mistakes people made.

Sometimes pattern can become an anti-pattern.

Each pattern will have its only notes with some 'official' description as well as my thoughts.
Sometimes i will add classification of some sort or any additional information i find interesting.

I will add patterns here when i have time between my studies

### Terms
1. SoC - Separation of Concerns, principle of dividing a system into distinct, independent sections, where each module addresses a specific, isolated responsibility or functionality

2. Anemic Class - an anti-pattern where classes act merely as data holders with properties, getters, and setters, but contain no business logic.

3. DI - Dependency injection, design pattern where an object receives its required dependencies from an external source rather than creating them itself.
